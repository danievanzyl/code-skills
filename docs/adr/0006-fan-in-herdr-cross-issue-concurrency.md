# feature-branch-fan-in-herdr v2: cross-issue concurrency over the sequential queue

## Context

[ADR-0005](0005-fan-in-herdr-sequential-queue.md) shipped `feature-branch-fan-in-herdr` v1 as a **sequential queue** — one full `orchestrator → runner → reviewer` pipeline at a time, squash-merged into the feature branch before the next starts — and explicitly deferred cross-issue concurrency to #61 "only once v1 has proven the sequential loop in practice." This ADR locks that v2 design ahead of the proof so it is ready to build when v1 proves out; it does not authorise implementation before then (#61 stays blocked by #53).

Two hard facts from v1 still bind v2:

1. **The Orchestrator is a single CLI-driven Claude session**, and herdr's `wait` API is **single-target-blocking** — both `wait output <pane>` and `wait agent-status <pane>` take one pane and block. There is **no wait-on-any-of-N** primitive. Watching N concurrent pipelines is therefore the core new mechanic ADR-0005 cited as the reason to defer.
2. **`feature-branch-fan-in` gotcha 5:** issues touching shared files **must** run sequential-with-merge-between. Parallelism is only ever legal for **verified-independent** issues. Concurrency must not reintroduce gotcha 5.

## Decision

**v2 adds cross-issue concurrency: parallel *across* proven-independent issues, sequential *within* any overlapping chain, bounded by a cap.** v1's sequential-with-merge-between remains the correctness floor everything degrades back to.

1. **Scheduling: per-issue event files funnelled through a signal pane to one blocking wait; bounded semaphore for launch.** herdr has no wait-on-any-of-N, so rather than round-robin busy-polling N tabs (which grows the single Orchestrator's context every sweep over a long queue — the same lean-context concern that drove v1's per-issue teardown), each pipeline appends its completion event to its **own per-issue file** — `~/.afk-herdr/<N>/events`, alongside ADR-0004's per-issue report dir, one line per event (`#N DONE|BLOCKED`). A **dedicated signal pane in its own tab** runs `tail -F` across the per-issue glob, funnelling every pipeline's events into one stream; the Orchestrator does a **single** `wait output` on that one signal pane and wakes on whichever pipeline reports next. Per-issue files (not one shared file) mean each pipeline is the **sole writer** of its own file — no multi-writer append contention — while the `tail -F` funnel preserves the single blocking wait. The Orchestrator processes exactly one event to completion (including merge + sync) before returning to the wait, which serialises all merges for free (decision 4). Launch is governed by a **bounded semaphore** of `K` permits; whenever a slot frees, the next eligible issue launches. Rejected: round-robin polling (context-growth tax on the single Orchestrator); it remains the fallback only if the signal-pane funnel proves unworkable in practice.

2. **Independence: static, conservative pre-partition; human-confirmed; operator override.** Before launching anything, the Orchestrator partitions the queue. Two issues may parallelise **only if** (a) neither declares a dependency on the other, transitively, **and** (b) their declared file-pointers/globs are disjoint. **Anything ambiguous** — an issue with no file pointers, a glob that *might* overlap — is treated as overlapping → same sequential chain. Fail safe, not fast. **Independence inputs come from two sources, both required for an issue to be parallel-eligible:** the **issue body** declares its file scope (paths/globs) and dependency signals (`Blocked by #N` / `Depends on #N`), and a **new label** (e.g. `parallel-eligible`) is an explicit opt-in — an issue **without** the label is never auto-parallelised, it defaults to the sequential chain regardless of what its body says. The label gates *eligibility*; the body supplies the *disjointness data*. This extends v1's "propose order, confirm" gate to "propose **partition + order**, confirm" before the first runner starts; **operator-supplied groups win outright** (mirroring v1's operator-order-wins). Rejected: optimistic-parallel-with-conflict-fallback (reintroduces gotcha-5 risk mid-flight for a wall-clock-only payoff ADR-0005 already said must never cost correctness); label-only or body-only (a label can't carry file globs; a body convention alone has no explicit opt-in and silently parallelises issues that merely happen to look disjoint).

3. **Concurrency cap: memory/`CLAUDE.md` default (3), operator arg overrides.** `K` is resolved exactly like v1's other run parameters (gh account, ready-for-agent label): memory or `CLAUDE.md`, ask if unknown; a `$ARGUMENTS` token overrides per-run. Default **3** — each pipeline is a worktree + two Opus-class pane agents, the Orchestrator is itself an Opus session tracking all of them, and 3 keeps the herdr UI legible (v1's teardown-for-legibility concern). `K` **is** the semaphore permit count (decision 1) and bounds *live* pipelines regardless of how wide the partition is: a 6-wide independent group under `K=3` runs 3 at a time.

4. **Merge: serialised through the completion channel; no mid-flight rebase; conflict → auto-resolve + partition-miss alarm.** The Orchestrator is the sole merger (as in v1); because it handles one completion event at a time (decision 1), only one merge is ever in flight — parallel pipelines *run* concurrently but *merge* one-at-a-time, squash-merge then `git pull --ff-only` its dedicated worktree, then release. **In-flight runners are never rebased** onto new feature-branch heads mid-flight — they keep their launch-time base, safe **because** decision 2 proved them disjoint; injecting a rebase into a running pane agent is exactly the fragile pane-mutation ADR-0004 avoids. A merge conflict between two supposedly-independent PRs is therefore **evidence the partition was wrong**, not a routine event. On conflict the Orchestrator invokes the `resolving-merge-conflicts` skill in its dedicated feature-branch worktree (it owns that worktree and is the sole merger), runs the repo checks per that skill, **notifies loudly** ("#X and #Y conflicted despite being classed independent — partition was wrong"), and **continues** — no hard pause. Exposing conflict markers for the skill requires the conflicting PR to be merged as a **local merge in the Orchestrator worktree**, not a bare server-side `gh pr merge --squash` (which just refuses a non-mergeable PR) — a real delta from v1's merge mechanic.

5. **Partition-distrust fallback: after N=2 conflicts, drain to fully-sequential.** One conflict is an anomaly; two indicts the partitioner. After the 2nd auto-resolved conflict the Orchestrator **stops taking new semaphore permits** and lets `K` drain to 1 — the remainder of the queue runs in v1's proven-correct fully-sequential mode. The degradation target already exists (it *is* v1), so this is "stop the intake valve," not new code.

6. **HITL: per-block notification, no auto-focus; block holds its permit.** With `K` live pipelines several can block at once. Attribution is free — the completion channel (decision 1) carries issue-`#N`-qualified BLOCKED events. Surfacing: fire **one herdr `notification` per blocked issue**; **do not auto-focus** (auto-focus with multiple blocks thrashes the human's focus). The human navigates at their own pace via the already-existing `issue #N` tab labels; the Orchestrator keeps a live blocked-set it can report on demand. A **blocked pipeline holds its permit** (it is live — worktree + panes — and a human unblocks it in-pane, resuming the pipeline). Consequence: `K` simultaneous blocks = a total stall on human attention — visible via the notifications, not a correctness issue. Rejected: a dedicated aggregated status tab (net-new UI surface the tab labels already cover — a possible later nicety); permit-release-on-block (a later unblock would then exceed `K` live pipelines).

7. **Terminal failure: drain-then-ask, queue-wide.** v1 halts the *whole* queue on terminal failure (reviewer `BLOCKED-needs-human`, unfixable red, human declines to unblock) because every downstream issue branches off the failed merge. Under concurrency that is too blunt — proven-independent in-flight siblings have no bearing on the failure. On terminal failure the Orchestrator **stops launching new pipelines, lets all in-flight ones drain and merge**, then presents the full picture (what merged with PR links, what failed and why, what never launched) and offers v1's same three choices, now queue-wide: intervene + resume / explicitly skip / stop and open a **partial** consolidated PR. Rejected: dependency-aware selective kill (fail only the failure's transitive dependents) — theoretically tighter but it bets the *live* kill decision on the same dependency graph decision 5 already hedges as fallible; drain-then-ask gets ~all the benefit (healthy in-flight work still lands) with no new runtime dependency-graph logic, and composes with the same "stop taking permits, drain" mechanic used in decisions 5 and 6.

## Consequences

- **New completion-channel contract.** Per-issue event files (`~/.afk-herdr/<N>/events`) + a `tail -F` **signal pane in its own tab** is a genuine new mechanic beyond ADR-0004's per-pipeline report files: each pipeline is sole writer of its own file (no append contention), and both the per-issue files and the signal-pane tab are torn down at Finish alongside the dedicated Orchestrator worktree. This is the single most load-bearing new surface in v2.
- **New eval hole:** on a conflict, decision 4 has the **Orchestrator author code** (the conflict resolution) that lands in the feature branch **unscored and unreviewed** — no Runner/Reviewer Trajectory, no Scorecard. Acceptable for pure reconciliation, but it widens ADR-0005's "the consolidated PR → main is not a Run" gap. Named here deliberately; closing it (scoring Orchestrator-authored merges) is out of scope for v2.
- **Semantic interference is caught only by the final integrated build/test.** Decision 4 keeps in-flight runners on their launch-time base, so a proven-independent sibling can merge *cleanly with zero textual conflict* yet leave another issue's code assuming behaviour that changed underneath it. v1's **required** final integrated build/test in the dedicated Orchestrator worktree is the only net for this — it stays required, never optional.
- **`CONTEXT.md` unchanged.** v2 adds workflow mechanics (completion channel, semaphore/permit, partition, distrust-fallback) — implementation vocabulary, not new domain roles or units. Same disposition as ADR-0005.
- **Queue-level Scorecard rollup stays deferred** to its own follow-up issue — orthogonal reporting, not concurrency; bundling it would dilute this ADR's focus.
- **The seven gotchas survive unchanged from v1** (ADR-0005's reconciliation table). Concurrency touches none of them directly: gotcha 5 is preserved by decision 2 (parallel only for proven-independent) with decisions 4b/5 as the tripwire + fallback when the proof is wrong; the rest are inherited via the shared `afk-issue-herdr` engine exactly as in v1.
- Ships as an **extension of the existing `feature-branch-fan-in-herdr` skill**, not a new skill — same outer loop, feature-branch targeting, and consolidated finish; v2 layers the partition, semaphore, shared channel, and drain-fallbacks onto v1's body. Stays **MIT** over the vendored **AGPL** `herdr` CLI, same posture as ADR-0003/0004/0005.

### Deferred-item disposition (#61)

| #61 deferred item | v2 disposition |
| ----------------- | -------------- |
| Cross-issue concurrency + configurable cap | Decisions 1–3 — shared channel + semaphore; cap = memory default 3, operator override. |
| Parallelizable-groups vs sequential-chain partition | Decision 2 — static conservative pre-partition, human-confirmed, operator override. |
| Orchestrator scheduling mechanic (interleave N poll loops) | Decision 1 — single blocking wait on a funnelled completion channel, **not** round-robin polling. |
| Merge-between vs parallel; re-sync so later work sees earlier merges | Decision 4 — serialised merges via the channel; no mid-flight rebase (safe by decision 2); disjointness means later work needn't see earlier merges until the final build. |
| Multi-simultaneously-blocked HITL | Decision 6 — per-block notification, no auto-focus, block holds permit. |
| Queue-level eval rollup | Deferred again to its own follow-up issue (reporting, not concurrency). |
